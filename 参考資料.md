# 梵(bon)— 参考資料

「誰でもプログラミングできる世界」を目指す bootable OS の設計メモ。 VRFSM の存在論をそのまま OS の中核に据える。

---

## 0\. 概要

- **名称**: 梵(Brahman)— 一切を貫く根本原理。梵我一如の梵。  
- **目的**: 「誰でもプログラミングできる世界」を、遍在する根本実在として実装する。  
- **中身**: VRFSM そのもの。古典 FSM(Playmaker 流)を計算の最小語彙とする。  
- **対応**: bootable / x86 / ARM 両対応。  
- **言語(実装)**: Rust。コアは `no_std` 級に小さく保つ。  
- **形式(保存)**: TOML。  
- **二古代文字**: 中国語(漢字)/ ブラーフミー(Brāhmī)。  
  - 漢字 \= 道家・仏教漢訳・東アジア哲学の語彙(運用・実装)  
  - ブラーフミー \= インド文字の祖、梵語/パーリ語を担う文字(「梵」に対応)  
  - 玄奘以来の梵漢併記の伝統を計算機環境に蘇らせる。  
  - ※ ブラーフミーの IME/フォントは未開拓領域 → 梵独自の貢献になりうる(実装フェーズの課題)。

## 1\. 開発方針

### 機材の役割分担

| 機材 | アーキ | 役割 |
| :---- | :---- | :---- |
| MacBook Pro M1 | ARM | 主開発機(当分メイン) |
| x86機(RTX 3050\) | x86\_64 | x86ターゲット検証、GPU/wgpu 実機確認 |
| MacBook Air M1 ("air") | ARM | 適宜・軽量検証 |

- ARM 2台 \+ x86 1台で、両ターゲットを実機で押さえられる理想的な布陣。  
- Mac でビルド → x86 機で焼いて起動確認、というサイクル。  
- ※ RTX 3050 機は OS 再インストール待ち。x86 bootable 検証に使うなら環境整備が必要。

### グラフィクス層

- **wgpu**(Rust の WebGPU 実装)を採用。  
- Metal(macOS)/ Vulkan(Linux)/ DX12(Windows)を自動抽象化 → macOS 開発・Linux 出力でコードはほぼ同一。VM 経由で Fedora を動かす必要性は薄い。  
- **ベアメタルでの wgpu は課題**: wgpu は OS の GPU ドライバに依存するため、自前 OS だとドライバから書く必要がある。  
  - 現実解: 当面 Linux カーネルをブートストラップとして借りる(方便、upāya)。完全ベアメタルは後段。

## 2\. 存在論(設計の核)— 思考の道筋

設計は一連の問いを降りていく形で結晶化した。以下はその到達点。

### 2.1 圏として考える

- **対象(object)**: 任意のモデルで表現可能(3D / 2D / 文字 / 何でも)。  
- **射(morphism)**: これも任意のモデルで表現可能。  
- 対象はアクションのリストを持つ(**空アクションも可**)。射もアクションを持つ。  
- OS の構成要素 \= 対象と辺(射)。  
- → これは圏の定義であり、同時に FSM の定義(状態=対象、遷移=射)。  
- **表現層を関手で分離**: 同じ圏(同じ計算)を、3D でも漢字でもブラーフミーでも音でも表現できる。  
  - 「誰でもプログラミングできる」の圏論的厳密化:関手 F: 梵圏 → 表現圏 が複数あり、ユーザーは表現圏を選べる。  
  - アクセシビリティの根本的再定義(視覚/聴覚/年齢/専門性に応じた函手的表現)。

### 2.2 古典 FSM \+ プロトコル分離

- **古典 FSM**(Playmaker 流): on\_enter / on\_stay / on\_exit、遷移時アクション。  
- メッセージパッシング的な並行性は**プロトコル層に押し出す** → コア語彙を最小に保つ(為道日損)。  
- プロトコル \= 「大域文脈の伝送」ではなく「局所文脈の貼り合わせ交渉」、最終的には「痕跡の投げ合い」。  
- VRFSM の P2P WebRTC 経験がそのまま活きる(中央サーバ=大域文脈を持たない)。

### 2.3 文脈はどこにあるか — 降りていった問い

1. **対象に閉じる?** → No。現実は状態をまたいで持続する文脈を持つ。  
2. **射が運ぶ / 環境が包む?** → Kleisli 圏的(射が文脈を運ぶ)が一旦の候補。  
3. **「文脈はどこにある? 互いに異なる文脈をそうだと仮定しているのが現実」** → 文脈に固定した居場所(自性)はない。各主体が「これが文脈だ」と\*\*仮設(prajñapti)\*\*し合っている。  
   - 縁起(pratītyasamutpāda)= Lacan の間主観性(ズレたまま噛み合う誤認の体系)= 層(sheaf)の発想。  
   - 層理論:大域文脈は最初から存在せず、ローカルな仮定が整合する限りで事後的に立ち現れる。  
   - cf. sheaf-theoretic contextuality(Abramsky ら)。  
4. **「整合などない」** → 大域切断(global section)は存在しない。**強文脈性**:局所は辻褄が合うのに大域には絶対に貼れない。  
   - Nāgārjuna:縁起すら空ずる。整合(関係)も実体化しない。戯論の寂滅。  
   - Lacan:「性関係は存在しない」。保証された整合の関係はない。  
   - 魯迅:希望(道)は先に在るのではなく、歩かれることで事後的に踏み跡が残るだけ。  
   - **実装的帰結**: 純粋メッセージパッシング・アクターモデル(Erlang/Elixir の最も純粋な形)。 共有メモリなし。send したものが意図通り受け取られた保証はない。受け手が自分の state で勝手に解釈する。

### 2.4 プラグマティズム — 二諦による解決

- 春丽はプラグマティストでもある。プラグマティズムは「対応としての真理」を捨て、「うまくいくこと(what works)」を真とする。  
- → 「整合などない」+「だが機能する」= **プラグマティックな縁起**。  
- 通信する理由:整合するからではなく、**送ることが機能するから**。真偽ではなく効き目(efficacy)。  
- cf. 食の哲学:見た目(対応・整合・儀礼)ではなく身体への効果(機能)。仏教の方便(upāya):効く筏。

**梵の二諦:**

勝義(底層) — 整合は無い。FSMは孤島。メッセージは痕跡。解釈は各自。

              共有状態を持たない。これは原理であり、曲げない。

世俗(方便層) — 機能する限りにおいて、整合"らしきもの"を道具として置く。

              同期、合意、共有ビュー —— すべて筏。効くから使う。真理だと思わない。

二層は矛盾しない。底層が原理、上層が道具。

プラグマティズム \= 二諦を行為で生きること。

- 整合系の道具(CRDT 等)は **opt-in のオプション**として分離。デフォルトは孤島、必要なら方便。

### 2.5 生成原理 — 全部が閉じた一手

**対象を生成するアクションがあればいい。**

- 対象は最初から在るのではない。**アクションが対象を産む**。生成も一つのアクション。  
- これで存在論の順序が反転し、別々に悩んでいた問題が一語彙に落ちる:  
  - **文脈問題が消える**:文脈が要るなら「文脈という対象を生成するアクション」を呼ぶ。置き場を持たない。縁起そのもの。  
  - **整合問題が消える**:出会いが新しい対象を生成するだけ(貼り合わせ gluing ではなく生成 generation)。  
  - **対象/射の二元論が消える**:射も「対象を生成するアクションの一種」。一つの生成原理に統合。  
- **原始操作は二つだけ**:**空為**(何もしない・恒等・無為)と **生為**(対象を生成する)。残りは全部その合成。  
  - cf. Lisp(atom と cons)、L-system、生成文法、Spivak のポリノミアル関手。  
  - cf. **行(saṅkhāra)**:十二縁起の第二支、生成するはたらき。仏教は二千年前に「アクションが対象を生成する」と言っていた。

### 2.6 始原 — 内部に持たない

- 「アクションが対象を産む」を無制限に許すと**生成の無限後退**(最初の生為は何の上で起きるのか)。  
- 候補:空(śūnya)から産む / 種(seed)を一つ置く / 自己生成(不動点)。  
- **春丽の答え**:そのどれでもない。**マウスタップ、または既存 TOML のインポート。**  
  - 始原はシステム内部に無い。**外から、人間の行為として来る**。  
  - 無限後退を内部の底(nil)で塞がず、人間の行為に開いた。始原は与えられるものではなく**為される**もの。  
  - 「触れたら始まる」→「誰でもプログラミングできる」が真に閉じる(前提知識ゼロの始原)。  
- **二つの始原 \= 二種類の人間への二つの門**:  
  - **タップ \= 無からの創造者**(創世。live editing の精神)。  
  - **TOML インポート \= 継承する者**(伝承。他者の生成履歴を、自分の梵で生成し直す。整合は保証されない=痕跡の解釈)。  
  - 個体発生と系統発生。

### 2.7 タップの操作フロー

タップ

  → 命名を促される(空名でもよい)

  → 対象が生まれる

  → その対象にアクションを詰める

- **タップ → 命名**:saṅkhāra(行)の次に来る **名色(nāma-rūpa)**\=名と形。生成直後に名が要請される。  
- **空名でもよい**:命名を促すが強制しない。  
  - 原理的:空為を原始に持つなら空名も当然許される(荘子「無名」、道徳経「名の名づくべきは常の名に非ず」)。命名すら方便。  
  - プラグマティック:創作では名前は後から来る。生成が命名に先行できる。  
- **アクションを詰める**:詰めるアクションの中にまた「生為」がある。  
  - → 生成されたものが、生成するものになる。縁起の輪。data がコードを宿し、コードが data を産む(Lisp 的)。

## 3\. 形式(TOML)

### 設計の核

- **TOML は「生成の履歴(レシピ)」であって「生成物のスナップショット」ではない。**  
- 保存すべきは結果の対象ではなく、それを産んだアクション(生為)の列。  
- 利点:TOML を読む \= 生成を追体験する / バージョン管理が自然(レシピの差分) / 食の哲学と一致(レシピであって完成写真ではない)。  
- 漢字キーは TOML ではクォートすれば任意 UTF-8 が使える(`"待機"`)。ベアキーは ASCII 限定。値は自由。

### スキーマ(最小)

\# 原始は二つだけ

\#   空為 — 何もしない(恒等・無為)

\#   生為 — 対象を生成する

\# タップで生まれた対象。名は空でもよい。

\[\[対象\]\]

名 \= ""                    \# 空名。後で詰めても、ずっと空でもよい

為 \= \[\]                    \# まだ空。これから詰める

\# アクションを詰めた対象

\[\[対象\]\]

名 \= "門番"

為 \= \[

  { 種 \= "空為" },                          \# 何もしない為も詰められる

  { 種 \= "生為", 産 \= { 名 \= "" } },         \# この対象が、無名の子を生む

  { 種 \= "生為", 産 \= { 名 \= "応対" } },

\]

## 4\. Rust コア(最小)

struct 対象 {

    名: Option\<String\>,   // None \= 空名。命名は促すが強制しない(型が約束する)

    為: Vec\<為\>,           // タップ直後は空。詰めていく

}

enum 為 {

    空,

    生(対象),              // 対象が対象を産む。再帰構造がここに宿る

}

// 始原は引数として外から注入される(内部に始原を持たない)

fn 梵(始原: Vec\<為\>) \-\> 世界 {

    始原.into\_iter().fold(世界::無(), |w, 為| w.産む(為))

}

// タップ      → 始原 \= vec\!\[為::生(空の対象)\]

// インポート  → 始原 \= toml::from\_str(...) で得た Vec\<為\>

- `名: Option<String>` の `None` が「空名でもよい」を型レベルで保証。  
- `生(対象)` が `対象` を含むことで再帰が立つ(無限に深くなれる構造が二つの enum 値だけで)。  
- `世界::無()` が唯一の内部的底だが、これは「空の Vec」であって哲学的な種ではない。無限後退は外側(人間の行為)で止まる。

### クレート構成案

梵 (bon)

├── bon-core/      \# 圏/FSM のコア型定義(no\_std 可能に保つ)

├── bon-toml/      \# TOML ⇄ 生為列(serde \+ toml)

├── bon-runtime/   \# 古典FSM実行エンジン(on\_enter/stay/exit)

├── bon-proto/     \# プロトコル層(将来のマシン間通信)

└── bon-render/    \# wgpu 表現層(後回し可)

- 最初は `bon-core` \+ `bon-toml` \+ `bon-runtime` で「TOML を読んで FSM を CLI で動かす」まで。表現層・ブート層は後段。  
- 方便層(整合っぽい道具)はフィーチャーフラグで後付け。コアは依存を持たない `no_std` 寄りに保つ。

## 5\. マイルストーン

M0: タップ一回で、無名の対象が一つ生まれ、画面に現れる

M1: 生まれた対象に名を入力できる(空のままでも置ける)

M2: その対象に「為」を一つ詰められる(まず空為でいい)

M3: 詰めた生為を実行すると、子の対象が生まれる

M4: 状態を TOML に書き出す(=生為の列として保存)

M5: TOML を読み込むと、生成が再生される

- M0〜M3 は GUI 無しで CLI / テストとして確認可能。`tap()` を関数呼び出しで模す。  
- GUI(wgpu / VRFSM 表現層)は M0 が通ってからで遅くない。  
- **始原がタップである以上、最初の一手は春丽自身が触れる方が梵らしい。**

## 7\. 留意点・未決事項

- **「新しい数学か」**:数学の部品(コアルジェブラ、層理論、ポリノミアル関手、アクターモデル)は既存。 新しいのは定理ではなく、**それらを一本に通す串(関手)の引き方**と、**整合の不在を出発点に据え、始原を外部化した実装思想**。 プラグマティックには「新しいかどうか」より「効くかどうか」が基準。  
- ブラーフミーの IME / フォント:現代 OS では未開拓。実装フェーズの課題かつ独自貢献の余地。  
- ベアメタル wgpu:GPU ドライバ問題。当面は Linux をブートストラップとして借りる。  
- 仏教存在論(縁起・空)と圏論的計算の対応を、比喩でなく実装まで降ろす — ここが春丽固有の領域。
