# 梵天OS — 計画.md

## このプロジェクトは何か

Asahi Linux上で動く、FSM(有限状態機械)と哲学を体験させるためのキオスク型単一作品。最終的にはM1 MacBook Air実機上のFedora Asahi Remixで、電源を入れたら直接梵天作品が立ち上がる体験を作る。

「梵天OS」と称しているが、独立したカーネルを書くわけではない。LinuxカーネルとAsahi/Mesa/Honeykrispのドライバスタックは借り、その上に独自のグラフィカルランタイムを置く。「OS」と呼ぶのは、ユーザーから見て他のアプリケーションが介在せず、画面全体を支配する単一実行体だから。

## 思想的背景(設計判断の根拠)

- **FSM-as-universal-model**: 過去のVRFSMプロジェクトおよびOppekepe VM構想の延長。すべての振る舞いを明示的な状態遷移として記述する。暗黙の制御フローを許さない。
- **縁起・無常**: 梵天という名前が示唆する「すべてを生み出す土台でありつつ、それ自身は固定された実体ではない」性質。状態間の遷移そのものが世界であり、状態は依存的に立ち現れる。
- **横の関係**: 縦社会的階層(systemd unit DAG、デスクトップ環境階層)を避け、フラットな状態空間を志向する。これが「キオスク単一作品」という形態の選択にも影響している。
- **VRFSMからの連続性**: WebGL/Three.jsで構築されたVRFSMの世界観をネイティブ環境に翻訳する。将来的にWebGPU版として再びWebに戻る可能性も視野に入れる。

## 技術選定とその理由

| 層 | 採用 | 理由 |
|---|---|---|
| ベースOS | Fedora Asahi Remix 43 (M1 Air実機) | GPU即動作の唯一の選択肢。Asahi+Honeykrisp公式。 |
| 開発環境 | UTM上のFedora 43 ARM64 VM (MacBook Pro M1ホスト) | Apple Hypervisor frameworkでネイティブ速度。実機と同じユーザランド。 |
| 言語 | Rust | Asahi自身がRust化方向。長時間連続動作のメモリ安全性。エコシステム(winit/wgpu/drm-rs)が揃う。 |
| グラフィックスAPI | wgpu | Vulkan/Metal/GLES/WebGPUを抽象化。将来のWeb版とソース共通化可能。Honeykrispが公式Vulkan 1.4対応。 |
| シェーダ言語 | WGSL | wgpu標準。GLSLからの移植コストは低い。 |
| ウィンドウ/入力(VM側) | winit 0.30 | Rustエコシステムの事実上標準。 |
| 画面取得(実機側) | DRM/KMS直接 + GBM + EGL/khronos-egl | Waylandコンポジタを介さない。`/dev/dri/card0`を直接掴む。 |

### 検討して却下した選択肢

- **C言語**: kmscubeなどサンプルは豊富だが、長時間連続動作の単一作品にはメモリ安全性の価値が大きい。
- **OpenGL ES (glow)**: VRFSMとの連続性は高いが、wgpuの方が長期的に有利。OpenGLは段階的にレガシー化。
- **ash (生Vulkan)**: 完全な制御は得られるが、単一作品のためにフレームバッファ・コマンドバッファを手書きする労力は釣り合わない。
- **独自Waylandコンポジタ**: wlroots/smithay上に書くことも検討したが、「他のクライアントを動かす」利点が単一作品では不要。中間抽象を介さずDRM/KMS直接が思想に合う。
- **カーネル空間コード(Rust for Linux/Asahiドライバ寄稿)**: Asahiカーネルへの寄与は別プロジェクトとして切り分け。今回の梵天OSはユーザ空間で完結する。

## アーキテクチャ

```
┌─────────────────────────────────────────┐
│ 梵天作品 (Rust)                          │
│  ┌─────────┐  ┌──────────┐              │
│  │ fsm.rs  │  │renderer.rs│             │  ← プラットフォーム非依存
│  └─────────┘  └──────────┘              │
│       ↑              ↑                   │
│       └──── main.rs ──┘                  │
│              ↓                           │
│  ┌──────────────────────────┐           │
│  │ platform trait           │           │
│  └──────────────────────────┘           │
│       ↓                ↓                 │
│  ┌──────────┐    ┌──────────┐          │
│  │winit_impl│    │ drm_impl │          │  ← 環境別実装
│  │  (VM)    │    │ (実機)   │          │
│  └──────────┘    └──────────┘          │
├─────────────────────────────────────────┤
│ wgpu (Vulkan/GLES/Metalバックエンド自動) │
├─────────────────────────────────────────┤
│ Mesa + Honeykrisp/Asahi GPU driver      │
├─────────────────────────────────────────┤
│ Linux + Asahi AGX kernel driver         │
└─────────────────────────────────────────┘
```

### 三層分離の原則

- `fsm.rs` は wgpu にも winit にも一切依存しない。純粋なRust。
- `renderer.rs` は FSM の内部構造を知らない。`world.state()`と`world.time_in_state()`のみ参照。
- `main.rs` は三者を糊付けするだけ。

この分離が守られている限り、レンダラ・プラットフォームを差し替えてもFSMコードは一文字も変わらない。

## 現状(v0で実装済み)

最小プロトタイプとして以下を実装済み。`bonten/`ディレクトリ参照。

### 実装済みの振る舞い

- winit + wgpuでウィンドウを開く(1280×720)
- 4状態のFSM: Void(無) → Arising(起) → Present(在) → Ceasing(滅) → Void...
- 状態に応じて画面クリア色を変える(藍黒・青・金・暗赤)
- 状態に入った瞬間に明るさパルスを入れる
- 入力: Space(進行)、R(リセット)、Esc(終了)
- 自動遷移: Arising→Present, Ceasing→Voidは2秒経過で自動

### ファイル一覧

```
bonten/
├── Cargo.toml          # winit 0.30, wgpu 22, pollster 0.4, env_logger 0.11
└── src/
    ├── main.rs         # ApplicationHandlerパターンのエントリポイント
    ├── fsm.rs          # State enum, World struct, 遷移ロジック
    └── renderer.rs     # wgpuのSurface/Device/Queue管理と描画
```

### 未実装(意図的に省いた)

実機での試行は反復が遅く、DRM masterを掴むコードはバグると画面が真っ暗になる。日常作業はVM、実機は統合検証のみ。

## VM環境セットアップ手順

UTMでARM64 Linux VMを作成する際の要点:

1. **Virtualization mode** を選ぶ(Emulationではない)。Apple Hypervisor framework経由でほぼネイティブ速度。
2. **virtio-gpu-gl** を有効化。UTMの「Display」設定で `virtio-ramfb-gl` か `virtio-gpu-gl-pci`。これが無いとwgpuがソフトウェアレンダリングに落ちる。
3. RAM 8GB以上、ストレージ 40GB以上推奨。
4. Fedora 43 ARM64 Workstation ISOを使う(実機Asahiに最も近いユーザランド)。

ゲストVM内のセットアップ:
```bash
# Rustツールチェイン
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source $HOME/.cargo/env

# wgpuがVulkanバックエンドを使うために必要
sudo dnf install vulkan-loader vulkan-tools mesa-vulkan-drivers

# winitの依存
sudo dnf install libxkbcommon-devel wayland-devel libX11-devel
sudo dnf install gcc pkg-config
```

動作確認:
```bash
cd bonten
RUST_LOG=info cargo run
```

## 次のタスク(Claude Codeへの作業指示)

優先順位順。一つずつ完了させること。各タスク完了後、動作確認できる状態に保つ。

### タスク1: プロジェクトを動かし、ビルドが通ることを確認

- `cargo build` でエラーなく通ること
- `cargo run` でウィンドウが開き、Spaceで色が変わることを確認
- もし依存バージョンの非互換でビルドエラーが出る場合、`Cargo.toml`を調整して通す。winit 0.30とwgpu 22のAPI差分には注意(winitは0.29→0.30でApplicationHandlerに大きく変わった)
- `cargo clippy --all-targets` で警告ゼロを目指す
- `cargo fmt` でフォーマット

### タスク2: シェーダパイプラインを足し、三角形を出す

現状の`renderer.rs`はクリアするだけで`draw_call`がない。最小の頂点+フラグメントシェーダ(WGSL)を書き、画面中央に三角形を出す。

要件:
- `src/shaders/triangle.wgsl` として外部ファイルに分離(`include_str!`で取り込む)
- 頂点はハードコード(3点)で良いが、`Vertex` struct と `wgpu::VertexBufferLayout` を正しく定義
- 三角形の色はFSMの状態に応じて変える(`renderer.rs`で uniform bufferを使うか、push constantsを使う。wgpu downlevel limitsだとpush constants不可なのでuniform bufferが安全)
- 既存のクリア色のFSM連動は維持する(背景=クリア色、前景=三角形)

### タスク3: platform抽象を実装に分離

現状`main.rs`の`App`が直接winitと結合している。これを抽象化する。

要件:
- `src/platform/mod.rs`に`Platform` traitを定義
- `Platform` trait は最低限以下のメソッドを持つ:
  - `fn run<F>(self, app_state: AppState, frame_fn: F)` のようなイベントループ起動メソッド
  - もしくは、より素直に `poll_events() -> Vec<InputEvent>`, `should_close() -> bool`, `present()`, `surface_size()` のポーリングAPI
- `src/platform/winit_impl.rs` に既存のwinit実装を移動
- `Renderer::new` のシグネチャから`Arc<Window>`への直接依存を取り除き、wgpu `Surface`を作るために必要な`SurfaceTarget`を抽象化する(`wgpu::SurfaceTargetUnsafe` を使うか、winit側で`Arc<Window>`を渡し続けるかは要検討)
- `cargo run` の振る舞いは変わらないこと(リファクタのみ)

設計の指針: VRFSMやVRChat的な3D空間を将来扱う可能性があるので、ポーリングAPI(`poll_events`)よりも、コールバックを渡す`run`型の方が適切かもしれない。ただし、winit 0.30のApplicationHandlerが既にコールバック型なので、それと馴染む形にする。

### タスク4: FSM定義をデータ駆動化する準備

現状`fsm.rs`の状態と遷移はRustコードに直接書かれている。これを将来、外部の宣言ファイル(.toml/.yaml/独自DSL)から読めるようにするための準備。

要件:
- `State` enum を文字列IDで識別できるようにする(`State::from_str`, `Display`実装)
- 遷移ルールを`Vec<TransitionRule>`のようなデータ構造で持つ
- ハードコードされた遷移を、データとして表現したものに置き換える
- 外部ファイル読み込みはこのタスクでは実装しない。**データ構造で表現できる形に書き換えるだけ**

### タスク5以降の候補(優先順位は後で決める)

- WGSLで三角形以上のジオメトリ(立方体、または球体)を出す
- テクスチャマッピング
- カメラ(view/projection matrix)
- 日本語・中国語・サンスクリットのテキスト描画(`glyphon`クレートが有力候補。WGSL対応のフォントレンダラ)
- 音声出力(`cpal` + `rodio`、またはより低レベルに`alsa-rs`)
- `drm_impl` の実装(実機モード)
- 状態の哲学的構造の本格設計(四聖諦・十二縁起・八正道・五蘊から選定)

## 哲学的構造の宿題(コード化前に決めること)

以下は丽さんが決めるべきこと。Claude Codeはここに踏み込まない。

1. **梵天作品が持つ状態の集合**: 現状の4状態(Void/Arising/Present/Ceasing)は仮置き。最終的に、四聖諦の4状態、十二縁起の12状態、八正道の8状態、道生一の生成的構造、五蘊の5、などのうちどれを採用するか、あるいは独自の構造を作るか。
2. **遷移の駆動源**: キー入力か、時間経過か、視線/カメラか、無条件遷移か、複数あるはず。それぞれの状態がどの種類の遷移を許すか。
3. **各状態が見せるもの**: 視覚・聴覚・テキスト。Three.js/VRFSMの既存リソースから何を引き継ぐか。
4. **作品の時間構造**: キオスクとして無限ループするのか、有限時間の体験(例: 30分で一周)か。誰もいないときの待機状態(attract mode)はあるか。
5. **入力デバイス**: キーボードのみか、マウスも使うか、視線(将来WebXR)、音声か。実機キオスクで何を置くかにもよる。

## 用語集

- **Asahi Linux**: Apple Silicon Macs用のLinux移植プロジェクト。M1/M2は安定、M3は実験的。
- **Fedora Asahi Remix**: AsahiコミュニティとFedora Projectの共同ディストリビューション。GPUアクセラレーション(Honeykrisp)が公式に動く唯一のディストロ。
- **Honeykrisp**: Asahi GPUドライバのMesa側実装名。Vulkan 1.4、OpenGL 4.6、OpenCL 3.0対応。
- **m1n1**: Asahi Linuxのブートローダ。Apple Silicon特有の初期化を行う。
- **DCP**: Display Coprocessor。Apple Siliconの画面出力を司るコプロセッサ。
- **AGX**: Apple GPUのアーキテクチャ名。PowerVR派生のタイルベースGPU。
- **DRM/KMS**: Direct Rendering Manager / Kernel Mode Setting。Linuxカーネルのグラフィックス基盤。Waylandを介さず直接画面を取る場合の入口。
- **wgpu**: WebGPU仕様準拠のRustクレート。複数バックエンドを抽象化。
- **WGSL**: WebGPU Shading Language。wgpuの標準シェーダ言語。
- **VRFSM**: 丽さんの既存プロジェクト。ブラウザ+WebXRで動く視覚的FSMエディタ。Three.js + Cloudflare Pages/Workers。
- **Oppekepe VM/OS**: 丽さんの構想する、FSMベースの計算モデル。梵天OSはその一つの実装。
